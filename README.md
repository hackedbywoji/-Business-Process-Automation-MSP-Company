# 🖨️ Print Consumables Automation — MSP Internal Project

This is a system I built from scratch while working as an IT technician at an MSP that handles managed print services for 70+ clients across Spain. The idea started simple: every time a printer ran low on toner, someone had to read an email, figure out who the client was, decide how to ship it, generate a delivery note manually, and notify the team. Multiple times a day. I automated all of it.

It's been running in production since June 2026 on a Windows 365 Cloud PC.

---

## 🎯 The Problem

The company uses MPS Monitor to track printer consumable levels. When a printer hits a low threshold, an automatic email lands in a shared corporate mailbox. From there, the manual process looked like this:

- Open Outlook, read the email
- Identify the client and check if they're in the courier database
- Decide: courier or local delivery van?
- Open the ERP, generate a delivery note
- Register the pickup with the courier
- Tell the team

For printers from a specific manufacturer (Brother), the workflow was different — they ship directly to the client, so instead of sending anything, someone had to log into the manufacturer's portal, search by serial number, check whether the consumable had been dispatched, and then close the alert in a separate management portal. If it was still "In Progress", they'd have to remember to check back in 2 hours.

This was happening 5-10 times a day. Lots of manual steps, easy to forget things, and completely dependent on someone being available and paying attention.

---

## ⚙️ What It Does Now

The system runs 24/7 and handles everything automatically. Here's the actual flow:

**Email monitoring**
A Python script polls the corporate mailbox every 5 minutes using the Microsoft Graph API. Access is set up through an Azure AD app registration (`Mail.Read` application permission, admin consent granted by the company admin). When a new email from MPS Monitor arrives with the subject "Solicitud consumibles", the script picks it up, extracts all the fields from the body (client name, model, serial number, consumable reference, page count), and decides what to do next.

**Standard consumables (Lexmark, OKI)**
The script fuzzy-matches the client name against a database of 70+ courier clients — handles abbreviations, missing words, typos. If there's a match above the confidence threshold, it goes by courier (1-page delivery note). If not, local delivery (2-page note). The PDF is generated by overlaying variable data on top of the original corporate template, pixel-perfect. Every processed consumable gets logged in a monthly Excel register with a sequential tracking code (`YY-MM-NNN`) that the admin team uses to import into the ERP. A Telegram notification goes to the team with all the details.

**Brother consumables**
These are detected automatically by a field in the email body (`VERIFICAR QUE HA MANDADO EL CONSUMIBLE BROTHER`) or by the printer model prefix (MFC-, DCP-, HL-). The script launches a headless Playwright browser, logs into the manufacturer's portal, strips the numeric prefix from the serial number (e.g. `E82414K4N662201` → `K4N662201`), searches for the device, and reads the consumable status. If it's "Shipped" or "Delivered", it then logs into the alert management portal, finds the right alert by serial number, and closes it. If it's "In Progress", it notifies the team to check back in 2 hours. If the primary portal is unreachable, it automatically falls back to a backup portal.

**Delivery note printing** *(in progress)*
The plan is to send the PDF to the office printer via email using the printer's built-in email printing feature (SMTP receive), connected over a VPN tunnel so the Cloud PC can reach the office network.

**Courier integration** *(in progress)*
Working on integrating ATENEA, the official Correos Express software, to automate pickup registration and label generation once the commercial credentials come through.

## 🛠️ Tech Used

Python, Playwright, Microsoft Graph API, Azure AD (app registration + admin consent), ReportLab + pypdf, OpenPyXL, thefuzz, Telegram Bot API, Windows 365 Cloud PC, Task Scheduler.

---

## 📚 What I Learned

I went into this knowing almost no Python. By the time it was in production, I'd built a custom email parser that handles both multiline and single-line formats (HTML-cleaned email bodies are tricky), a PDF overlay system calibrated to the exact pixel coordinates of the corporate template, a fuzzy matching pipeline that correctly identifies clients even when the name comes through mangled, and a headless browser automation that works around anti-bot detection on external portals.

The thing I'm most proud of is that it just runs. No one has to think about it.
